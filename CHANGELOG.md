- **2026-07-06 ‚Äî Fix: `CoreUtils._prepare_reparent`'s childless-parent guard was comparing full-DAG-path children against a node set built from the caller's raw (possibly short-name) input ‚Äî a format mismatch that, depending on which side happened to be short vs full, silently skipped creating the protective temp-null in one of the two input styles.** Surfaced by the previous entry's `combine_non_instanced` default: `AutoInstancer._combine_non_instanced` always resolves candidates to full DAG paths before calling `EditUtils.combine_objects`, and when it consumed every child of a plain (non-tagged) group, Maya's `polyUnite(..., ch=False)` deleted the now-emptied parent before `_finalize_reparent` could re-parent the combined mesh back into it ‚Äî reproduced as `test_original_mesh_separated`'s `IndexError` (the test's group vanished mid-run) once the full mayatk suite ran end-to-end. Root cause: `_prepare_reparent` queried the candidate parent's `children` WITHOUT `fullPath=True` while its own `node_set` held whatever format the caller passed, so the "would this parent go empty" membership check (`children` vs `node_set`) only ever matched when both sides happened to already agree ‚Äî true for `test_combine_preserves_parent_group`'s short-name-only case (by accident, not by design), false for the full-path case `_combine_non_instanced` actually uses. Fix: both sides now resolve to full paths (`node_set = set(cmds.ls(node_strs, long=True) or node_strs)`, `children` queried with `fullPath=True`) ‚Äî format-agnostic regardless of what the caller passes. New regression test `test_combine_preserves_parent_group_full_paths` pins the previously-untested direction. `test_original_mesh_separated.py`'s prototype cube was also bumped to `sx=sy=sz=6` (432 triangles, above `MICRO_TRI_THRESHOLD`) ‚Äî its plain 12-triangle cube was legitimately being routed to the (correct, newly-enforced) micro-mesh combine path rather than the instancing path the test's docstring says it verifies; a stale test assumption, not a second bug. **Follow-up critique found the same function only ever protected the FIRST parent it saw going empty** (`break` after the first temp-null) ‚Äî a combine spanning candidates from multiple original groups (exactly what `_combine_non_instanced` does across a whole scene) left every OTHER emptied group completely unprotected; empirically reproduced (two 2-cube groups, different materials ‚Üí the second group silently deleted) and fixed: the loop no longer `break`s, and since multiple temp-nulls can now coexist under different parents with the same literal name (Maya only disambiguates same-parent siblings), each is tracked by UUID (`cmds.ls(name, uuid=True)`, the same survives-rename/reparent pattern `_auto_instancer.py`'s `InstanceCandidate` already uses) so cleanup can never delete the wrong one or hit an ambiguous-name error. **Left explicitly out of scope:** `_finalize_reparent` still re-parents every result from a multi-cluster combine (`group_by_material`/`cluster_by_distance` producing >1 combined mesh) under the SAME single captured parent (the first input's) ‚Äî in the reproduction above, the second group's combined mesh lands under the first group's location rather than its own (now-also-preserved) group. Also fixed while verifying this: `run_tests.py`'s results file (`test/temp_tests/test_results.txt`) was a single fixed path shared by every `MayaTestRunner` instance regardless of Maya port ‚Äî running a second, differently-ported `run_tests.py` invocation while the `--all` background suite was still in flight `unlink()`-and-rewrote the SAME file, and `wait_for_results`'s file-based fallback (which just checks for a `"SUMMARY"` marker) then reported the ORIGINAL long-running `--all` run "complete" with the SECOND, unrelated run's small result ‚Äî silently corrupting the exact full-suite badge this pass needed. - **New tool ‚Äî Curve to Tube (`nurbs_utils/curve_to_tube.py`).** Sweeps a circular profile along selected NURBS curve(s) into a tube, output as a **NURBS surface** or a **polygon mesh** (the `CurveToTubeSlots` panel's `cmb000` picks the type and greys out the options that don't apply). `CurveToTube.create(curves, output_type, radius, sections, path_divisions, degree, caps, quads, live, cleanup)` is pure geometry (classmethods returning node names), callable straight from a script. **NURBS** path extrudes a `circle` along the curve (`extrudeType=2`, component-pivot + fixed-path + profile-normal); it **self-conforms outward** by reversing the U direction when a mid-path ring's normal points inward (the extrude could emit either orientation depending on how the curve was drawn ‚Äî it was coming out inverted) and sets a fine **display smoothness** (the viewport `3` preset) so the surface renders smooth, not faceted. **Polygon** path places rings by **RDP curvature simplification**: the curve is densely sampled and run through `pythontk.MathUtils.simplify_rdp` (Ramer-Douglas-Peucker), which keeps points where the path bends and drops near-collinear runs. Live and baked share the same topology. All paths are `cmds`-based, so a committed tube is a single clean Ctrl+Z; a closed/periodic curve makes a watertight torus.
